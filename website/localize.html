Questions

What does "cultural context" mean in practice?＋

Every translated output includes a per-language annotation block that flags: which date format to use (DD/MM/YYYY vs MM/DD/YYYY vs YYYY年MM月DD日), local currency symbol placement (€1,200 vs 1.200 €), idioms that don't carry across cultures, punctuation norms (em-dash usage, exclamation frequency), and register notes (formal usted vs informal tú in Spanish, honorific levels in Japanese). You get the translation plus the reasoning — so your team can QA it confidently.

How does RTL layout work?＋

Arabic (AR) and Hebrew (HE) outputs include Unicode right-to-left markers and a culture note flagging which UI elements need to be mirrored — navigation, image placement, text-align, flex direction. The translated text is ready to drop into an RTL-capable CMS or email template. The culture note tells your developer exactly what CSS changes (dir="rtl", text-align: right, etc.) are needed before shipping.

Can I specify a tone — formal vs casual?＋

Yes. Set the tone field in your input JSON to formal, neutral, or casual. This combines with the purpose field (marketing, technical, or conversational) to control register. For example, purpose: marketing, tone: casual in Spanish produces tú-form copy with punchy CTA verbs; purpose: technical, tone: formal produces usted-form documentation language with precise terminology.

What happens if the API quota is exceeded?＋

If a translation call fails — quota, rate limit, or network error — Localize writes a [NEEDS_TRANSLATION_TO_X] marker file in the output folder with the failure reason logged inside. Nothing is silently skipped. The marker file tells you exactly which language failed and why, so you can rerun just that target without reprocessing the entire batch. The failure is also written to the agent's run log for Batman to sweep up.
